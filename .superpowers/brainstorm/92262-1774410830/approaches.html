How should hooks communicate with the TUI?

This is the core architectural decision — how event data flows from Claude Code into the visualizer

A

Unix Domain Socket

TUI listens on a UDS. Hooks are tiny shell scripts that pipe stdin JSON to the socket via nc or socat.

Pros

  • Zero latency, no serialization overhead
  • No port conflicts — uses a file path, not a network port
  • Simple hook scripts (one-liners)
  • Natural fit for local-only tool

Cons

  • Requires nc or socat on the system
  • Slightly harder to debug than HTTP

⭐ Recommended — simplest, fastest, most Unix-native approach

B

Local HTTP Server

TUI runs an HTTP server on localhost. Hooks POST JSON to http://localhost:PORT/events.

Pros

  • Easy to debug (curl, browser dev tools)
  • Hooks use curl which is universally available
  • Could later add a web UI alongside the TUI

Cons

  • Port allocation — need to pick/discover a free port
  • Slightly more overhead than UDS
  • More code to handle HTTP parsing
C

File-based (append-only log)

Hooks append JSON lines to a file. TUI tails the file (like tail -f).

Pros

  • Simplest possible hooks — just cat >> file
  • No server component needed
  • Built-in persistence for free

Cons

  • Polling introduces latency (or need inotify/kqueue)
  • File locking concerns with concurrent writers
  • Cleanup/rotation needed for long sessions
